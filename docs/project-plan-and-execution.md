# 富文本编辑器项目计划与阶段执行汇报

## 1. 项目概述

本项目定位为面向中长期演进的富文本编辑器前端基础工程，目标是构建一套具备良好扩展性、可承载复杂文档能力、并可逐步演进为“飞书风格文档产品”的技术底座。

截至当前阶段，项目已完成基础工程搭建、编辑器核心能力接入、协同能力依赖预埋、构建与校验闭环，以及阶段性文档沉淀，具备继续进入产品化能力扩展的基础条件。

## 2. 建设目标

本项目当前阶段的建设目标如下：

- 建立可持续扩展的富文本编辑器基础架构。
- 输出可运行的编辑器演示版本，作为后续功能扩展与业务评审载体。
- 为协同编辑、评论、权限控制、版本管理、导入导出和 AI 辅助能力预留架构空间。
- 通过文档化方式沉淀技术路线、实施节奏和能力边界，降低后续建设成本与沟通成本。

## 3. 技术路线与选型说明

### 3.1 核心技术栈

- 前端框架：Next.js 16 + React 19 + TypeScript
- 编辑器内核：Tiptap 3
- 协同层预留：Yjs
- 协同传输预留：Hocuspocus Provider
- 样式体系：Tailwind CSS v4
- 辅助能力：Radix UI、Floating UI、Zustand

### 3.2 选型依据

- Next.js App Router 便于后续承接服务端能力、权限控制与多文档路由结构。
- Tiptap 构建于 ProseMirror 之上，扩展能力强，适合实现复杂富文本交互。
- Yjs 是当前成熟度较高的协同编辑 CRDT 方案，适用于多人实时编辑、离线恢复与冲突处理场景。
- Tailwind CSS v4 有利于快速推进样式开发，并支持后续组件体系化治理。

## 4. 分阶段实施规划

### 第一阶段：编辑器基础能力建设

- 完成段落、标题、列表、任务列表、引用、代码块等基础块结构。
- 完成加粗、斜体、下划线、高亮、链接等基础行内样式。
- 完成表格等常见内容块的基础支撑。
- 形成可运行的编辑器演示版本。

### 第二阶段：富文本定制化能力建设

- 完成 Word 内容粘贴兼容与图片粘贴能力。
- 完成链接功能整理与场景化演示。
- 完成“左图右文”模板块能力。
- 完成表格合并、拆分、插图等高级能力验证。

### 第三阶段：协同与业务化能力建设

- 接入共享文档模型与协同状态管理。
- 增加光标感知、在线状态、房间会话与鉴权控制。
- 逐步完善评论、版本历史、导入导出和附件链路。

### 第四阶段：产品化增强能力建设

- 完成建议模式与评论体系。
- 完成版本恢复与操作审计能力。
- 扩展 AI 辅助编辑能力，如总结、改写、润色与翻译。

## 6. 执行过程说明

### 6.1 需求澄清与范围确认

项目启动阶段首先明确本项目并非一次性展示型 Demo，而是面向后续持续扩展的编辑器能力基础工程。基于这一前提，技术路线不以短期视觉呈现为唯一目标，而是优先考虑可扩展性、可维护性与后续接入协同和业务能力的可行性。

### 6.2 工程初始化与基础搭建

在当前工作区完成 Next.js 工程初始化，并采用 TypeScript 作为主开发语言，确保后续复杂编辑器状态、扩展节点和插件能力具备良好的类型约束。

### 6.3 编辑器核心能力接入

通过 Tiptap 完成编辑器实例初始化，接入标题、列表、表格、链接、代码块等基础能力，并将编辑器渲染至首页，形成可运行、可验证的实验版本。

### 6.4 协同能力预留

当前阶段已安装 Yjs 与 Hocuspocus Provider 相关依赖，但未在演示版本中强行接入未完成设计的协同后端，以保证当前建设节奏聚焦于基础体验与可视化验证。

### 6.5 第二阶段定制化需求落地

围绕业务提出的四项重点需求，新增独立的富文本实验组件与说明文档，完成以下方向的初步实现与演示：

- Word 内容粘贴结构保留与样式清洗
- 图片直接粘贴与上传插入
- 链接功能场景化示例
- 左图右文模板块
- 表格合并、拆分、增行增列与表格插图能力

### 6.6 校验与收口

所有新增能力均已在当前代码基线上完成静态校验与生产构建验证，确保本阶段产出具备继续演进与对外演示的基础稳定性。

## 7. 第二阶段需求分解：富文本定制化产品

第二阶段目标是将编辑器从“具备基础编辑能力”推进到“具备产品化验证价值”的状态，重点围绕四类定制化需求开展调研、实现、展示与文档沉淀。

### 7.1 需求一：Word 内容复制可用，不丢格式，支持图片粘贴

#### 需求目标

- 支持将 Word 文档中的内容复制至编辑器，并尽可能保留标题、段落、加粗、列表、表格、链接等主要格式。
- 支持用户直接将截图或图片粘贴到编辑器中，降低内容录入成本。

#### 调研结论

- Word 粘贴通常伴随大量 Office 专有 HTML 与冗余样式。
- 实际业务中优先保留的是内容结构和主要格式，而非 Word 页面级版式。
- Tiptap 可通过粘贴事件钩子与 HTML 转换机制介入处理流程。
- 图片粘贴可通过读取剪贴板中的图片文件并转为图片节点方式实现。

#### 实施方向

- 增加 Word 粘贴识别与清洗策略。
- 保留标题、列表、表格、链接等结构化标签。
- 识别并插入图片文件。
- 后续可扩展为“上传后回填 URL”的正式媒体链路。

#### 风险与边界

- 当前目标不包括对页眉页脚、批注、文本框、分页控制等 Word 特性的完整还原。
- 对外表述应明确为“保留业务可用文档结构与主要格式”。

### 7.2 需求二：富文本基础功能与链接功能调研

#### 需求目标

- 对富文本基础能力进行统一梳理。
- 对链接功能进行单独说明，以便作为通用能力标准化输出。

#### 基础功能范围

- 标题、段落、引用、代码块
- 有序列表、无序列表、任务列表
- 加粗、斜体、下划线、高亮
- 图片、表格、链接
- 基础对齐与常见块结构能力

#### 链接能力结论

- 选中文本后插入链接是基础交互形态。
- 自动补全 `https://` 可降低输入门槛。
- 外链默认新窗口打开更符合业务使用习惯。
- 后续可进一步扩展为外部链接、站内文档链接和业务对象链接。

### 7.3 需求三：左图右文

#### 需求目标

- 支持富文本中的图文混排能力。
- 能力形态需具备模板化、可重复使用的特征，而非一次性拼装式排版。

#### 调研结论

- 普通图片加段落的方式在复制、移动和复用场景下稳定性较差。
- 更优解是将其定义为自定义块或模板块，确保版式与内容保持为统一结构单元。
- 该能力适用于功能说明、知识卡片、公告区块、案例介绍和 SOP 模板。

#### 实施方向

- 定义“左图右文”自定义块。
- 支持标题、描述、图片地址、图片说明、风格主题等属性。
- 后续可扩展方向包括图文互换、样式切换和拖拽排序。

### 7.4 需求四：表格合并、拆分、插入图片

#### 需求目标

- 在基础表格能力之上，补齐合并单元格、拆分单元格和表格内插图能力。

#### 调研结论

- 表格高级能力是需求文档、审批文档、知识库和方案文档中的高频需求。
- Tiptap 所基于的 ProseMirror Table 体系已具备合并、拆分、增删行列等基础能力。
- 表格插图的关键在于允许图片节点进入单元格内容，而非单独开发另一套上传表格组件。

#### 实施方向

- 增加表格操作区，支持合并、拆分、增行、增列和删除表格。
- 保证图片插入逻辑兼容单元格内容区域。
- 后续可进一步扩展表头样式、背景色与更复杂的跨行跨列配置。
